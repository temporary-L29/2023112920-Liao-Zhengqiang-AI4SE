# 实验六各方案 Prompt 中文记录

本文档记录实验六实际调用 DeepSeek 时使用的 prompt 结构，并说明每个占位符在运行时嵌入了什么信息。

源码依据：

- `experiment6/src/improved_prompt_templates.py`
- `experiment6/src/enhanced_context_builder.py`
- `experiment6/src/config.py`

## 一、整体调用结构

每次 API 调用由两条 message 组成：

```text
[
  {"role": "system", "content": <对应 P5/P6/P7/P8 的 system prompt>},
  {"role": "user", "content": <对应 P5/P6/P7/P8 的 user prompt>}
]
```

其中 user prompt 中有三个通用嵌入块：

| 占位符 | 实际嵌入内容 |
| --- | --- |
| `{few_shot}` | 4 个平衡示例：2 个 merged、2 个 not_merged；包含标题、正文、改动文件、风险清单、分析、merge_probability、unmerged_risk_score、risk_factors |
| `{context}` | 当前 PR 的增强上下文；不同方案嵌入 C5、C6 或 C8 |
| `{output_format}` | 统一 JSON 输出格式，要求只输出合法 JSON，包含 merge_prediction、merge_probability、unmerged_risk_score、confidence、risk_factors、evidence_summary、review_comments |

## 二、共享嵌入块

### 2.1 `{few_shot}`：平衡示例

该块用于给模型提供“merged 与 not_merged 都可能合理”的示例，避免实验五中默认预测 merged 的倾向。

嵌入内容翻译如下：

```text
## 平衡分析示例

### 示例 1：Merged — 干净、有测试覆盖的修改
标题：Fix copilot inline suggestion flickering
正文：修复 suggestion provider 中的竞态条件。根因是快速输入时 debounce timer 没有被重置。新增了覆盖该竞态条件的单元测试。
改动文件：2 个文件（1 个源码文件，1 个测试文件）
风险清单：有测试、小范围修改（+15/-3）、描述清晰
分析：低风险 bugfix，带测试覆盖。高置信度可合并。
merge_prediction：merged
merge_probability：0.92
unmerged_risk_score：0.08
Risk Factors：[]

### 示例 2：NOT Merged — 大范围、未测试的 AI 生成修改
标题：WIP: refactor scheduler using GPT-generated code
正文：我使用 ChatGPT 生成了新的 scheduler 实现。仍然需要测试。
改动文件：12 个文件，+850/-120 行
风险清单：缺少测试、大范围修改（12 个文件）、大量新增（+850）、AI 工具痕迹：gpt、核心模块、描述过短
分析：大范围、未测试的核心 scheduler 重构。明确说明使用 AI 生成。缺少测试且动机不清晰。被拒绝风险高。
merge_prediction：not_merged
merge_probability：0.15
unmerged_risk_score：0.85
Risk Factors：["大范围重构缺少测试", "AI 生成代码触及核心模块", "缺少迁移计划或 benchmark", "PR 描述缺少充分理由"]

### 示例 3：Merged — 机械式类型标注
标题：Add type hints to DataFrame.groupby (generated with Copilot)
正文：使用 GitHub Copilot 为 groupby 模块添加缺失的类型标注。所有现有测试通过。没有行为变化。
改动文件：3 个文件，+45/-12 行
风险清单：有现有测试套件、小范围修改、AI 工具痕迹：copilot（但属于机械式修改）
分析：机械式类型标注，虽然使用 AI 工具，但语义风险低。现有测试提供覆盖。
merge_prediction：merged
merge_probability：0.85
unmerged_risk_score：0.15
Risk Factors：["使用了 AI 工具，但改动机械且测试充分"]

### 示例 4：NOT Merged — 破坏性 API 改动且没有文档
标题：Add new attention mechanism (Claude-assisted implementation)
正文：Claude 帮助实现了新的 attention mechanism。对现有 API 有破坏性变更。
改动文件：8 个文件，+420/-80 行
风险清单：缺少测试、核心模块、API 改动没有文档、AI 工具痕迹：claude
分析：破坏性 API 改动缺少迁移计划。没有 benchmark 或测试。AI 辅助实现新算法，需要仔细审查。
merge_prediction：not_merged
merge_probability：0.20
unmerged_risk_score：0.80
Risk Factors：["破坏性 API 改动没有迁移指南", "新算法缺少 benchmark", "AI 生成的新实现存在正确性不确定性", "没有文档更新"]
```

### 2.2 `{output_format}`：统一输出格式

该块强制模型只输出 JSON，不允许输出 Markdown 或额外解释。

嵌入内容翻译如下：

```text
## 输出格式
你必须只返回一个合法 JSON 对象（不要 markdown，不要额外文本）：

{
  "merge_prediction": "merged" 或 "not_merged",
  "merge_probability": <0.0 到 1.0 的浮点数>,
  "unmerged_risk_score": <0.0 到 1.0 的浮点数，越高表示越可能不会合并>,
  "confidence": "low" 或 "medium" 或 "high",
  "risk_factors": ["<具体风险 1>", "<具体风险 2>", ...],
  "evidence_summary": "<一句话总结关键推理依据>",
  "review_comments": [
    {
      "file": "<文件路径或 'general'>",
      "line": null,
      "severity": "nit|minor|major|blocker",
      "comment": "<可执行的审查意见，要具体说明需要修什么>"
    }
  ]
}

重要：
- 不要默认预测 "merged"。很多 PR，尤其是 AI 生成代码 PR，确实不会被合并。
- 寻找 PR 可能被拒绝的具体理由：缺少测试、范围过大、破坏性变更、描述不足、AI 痕迹。
- review_comments 必须可执行：要提到具体文件、函数或行为。
- 如果 risk_factors 为空，要在 evidence_summary 中解释原因。
```

## 三、`{context}` 中嵌入的信息

### 3.1 基础 PR 上下文：所有 C5/C6/C8 都包含

基础上下文来自当前 PR 的早期可用信息，不包含当前 PR 的真实合并结果或真实 review 结果。

嵌入字段如下：

| 信息类别 | 具体嵌入内容 |
| --- | --- |
| 仓库信息 | Repository、PR ID |
| 改动文件列表 | filename、status、additions、deletions |
| 文件类型标记 | 如果文件名命中 test/spec/tests 等模式，标记 `[TEST]`；如果命中 json/yaml/toml/lock/package/CI 等模式，标记 `[CONFIG]` |
| 新增代码行 | 从 patch 中抽取新增行，最多 200 行；新增代码文本超过 6000 字符会截断 |
| PR 描述 | title、body；body 最多 3000 字符 |
| commit 信息 | num_commits、num_changed_files、total_additions、total_deletions、commit_messages；commit messages 最多 1500 字符 |

### 3.2 C5：Risk Checklist Context

C5 = 基础 PR 上下文 + 自动风险清单。

额外嵌入信息如下：

| 风险/观察项 | 具体判断依据 |
| --- | --- |
| 是否缺少测试 | 改动文件中是否没有 test/spec/tests 等测试文件 |
| PR 改动文件数量 | `num_changed_files`；超过 10 个文件标记为 LARGE_SCOPE，超过 5 个文件标记为中等范围 |
| PR 新增代码行数 | `total_additions`；超过 500 行标记为 LARGE_ADDITIONS，超过 200 行标记为中等新增量 |
| 配置/CI/依赖变更 | 文件名是否命中 `.json`、`.yaml`、`.toml`、lock 文件、package、requirements、Dockerfile、GitHub Actions、CI 等模式 |
| 核心模块/API 文件 | 文件路径是否包含 public、api、export、interface、abstract、core、main、init、index 等关键词 |
| PR 描述长度 | body 长度小于 100 字符标记为 SHORT_DESCRIPTION，小于 300 字符标记为描述较短 |
| AI 工具痕迹 | title/body 中是否包含 copilot、chatgpt、gpt、claude、cursor、codex、deepseek、qwen、generated by、ai-generated、code agent 等关键词 |
| API 改动无文档 | patch 中出现 public/export 且没有 doc/readme/changelog/md/rst 等文档文件 |
| commit 中的 AI 痕迹 | commit_messages 中是否包含 generated、copilot、ai |
| 数据集 AI 标记 | `has_ai_generated_code` 是否为 true |

注意：C5 只给观察事实，不直接嵌入当前 PR 的 merge label。

### 3.3 C6：Repository Policy Context

C6 = C5 + 仓库级历史策略信息。

额外嵌入信息如下：

| 信息类别 | 具体嵌入内容 |
| --- | --- |
| 同仓库历史 AI PR 数量 | 从实验五完整 AI PR 数据集中筛选同一 repo，并排除当前 PR |
| 同仓库历史 AI PR 合并率 | `same_repo["is_merged_bool"].mean()`，使用 leave-one-out，排除当前 PR |
| 同仓库 AI PR review 覆盖率 | 有非空 review_comments_text 的历史 PR 占比 |
| 未合并 PR 常见关键词 | 在同仓库未合并历史 PR 的 title 中统计 test、documentation、breaking、WIP、refactor、draft、dependency、bump、fail、bug、fix、feature 等关键词 |
| AI policy 迹象 | 历史 review_comments_text 中是否出现 ai-generated、ai policy、no ai、ban ai、ai contribution、generated code、llm、chatgpt、copilot policy 等关键词 |
| review 风格 | 平均 review comment 长度；reviewer 是 actively 还是 selectively comment |
| 仓库低合并率提示 | 如果同仓库 AI PR 历史合并率低于 50%，提示该仓库对 AI PR 审查更严格 |

注意：C6 使用历史 PR 的结果与评论，但排除当前目标 PR，避免泄露目标 PR 结果。

### 3.4 C8：Full Improved Context

C8 = 基础 PR 上下文 + C5 + C6 + C7。

C7 额外嵌入 3 条历史相似 PR：

| 信息类别 | 具体嵌入内容 |
| --- | --- |
| 相似 PR 检索范围 | 实验五完整 AI PR 数据集，排除当前 PR |
| 相似度依据 | 同仓库优先、改动文件数量相近、total_additions 相近、title 关键词重叠 |
| 每条相似 PR 内容 | title、repository、num_changed_files、additions/deletions、历史 outcome、review summary |
| 历史 outcome | 相似历史 PR 的 merged/not_merged 结果，用作参考 |
| 说明文字 | 提醒模型历史结果仅供参考，最终判断必须基于当前 PR 内容 |

C8 有长度控制：最大上下文约 16000 tokens。超限时优先保留基础 PR 信息、风险清单和仓库策略，再截断。

## 四、M1：P5 AI-Risk-Aware + C5 Risk Checklist

### 4.1 方案嵌入说明

| 部分 | 内容 |
| --- | --- |
| Prompt 类型 | P5：AI-Risk-Aware |
| Context 类型 | C5：基础 PR 上下文 + AI 代码风险清单 |
| 主要目的 | 让模型主动寻找 AI 生成代码可能不会合并的风险 |
| 嵌入的 PR 信息 | 仓库、PR ID、改动文件、PR 长度/规模、代码新增行、PR 标题/正文、commit 数量和信息 |
| 嵌入的风险信息 | 缺测试、改动范围、PR 新增行数、配置/依赖变更、核心/API 文件、描述长度、AI 工具痕迹、API 改动无文档 |

### 4.2 System Prompt 中文翻译

```text
你是一名经验丰富的代码审查者，专门审查 AI 生成代码。

你的首要任务是识别 AI 生成 Pull Request 中可能导致它们不会被合并的风险。

关键原则：
1. AI 生成代码经常存在细微问题：幻觉 API、遗漏边界情况、过度工程化方案。
2. 寻找结构性问题：缺少测试、范围过大、破坏性变更、文档不足。
3. 很多 AI 生成 PR 确实会被拒绝，不要默认预测 "merged"。
4. 你的审查意见必须可执行，要引用具体文件、函数或模式。
5. 提供的风险清单只是指导，不是标签。你要形成自己的判断。
```

### 4.3 User Prompt 中文翻译

```text
你正在审查一个可能包含 AI 生成代码的 Pull Request。

{few_shot}

---

现在分析下面这个 Pull Request：

{context}

---

{output_format}

记住：寻找这个 PR 可能不会被合并的理由。要具体，并基于证据。
```

## 五、M2：P6 Contrastive + C5 Risk Checklist

### 5.1 方案嵌入说明

| 部分 | 内容 |
| --- | --- |
| Prompt 类型 | P6：Contrastive |
| Context 类型 | C5：基础 PR 上下文 + AI 代码风险清单 |
| 主要目的 | 强制模型同时比较“支持合并”和“反对合并”的证据，降低默认乐观偏差 |
| 嵌入的 PR 信息 | 仓库、PR ID、改动文件、PR 长度/规模、代码新增行、PR 标题/正文、commit 数量和信息 |
| 嵌入的风险信息 | 与 M1 相同：缺测试、改动范围、PR 新增行数、配置/依赖变更、核心/API 文件、描述长度、AI 工具痕迹、API 改动无文档 |

### 5.2 System Prompt 中文翻译

```text
你是一名经验丰富的代码审查者，正在进行平衡、基于证据的审查。

关键要求：你必须明确列出支持合并和反对合并的证据，然后再做判断。
不要因为修改看起来小或简单就默认预测 "merged"。

流程：
1. 列出支持合并的证据（merge_evidence）
2. 列出反对合并的证据（reject_evidence）
3. 权衡双方
4. 如果证据平衡或不确定，倾向于给出更低的 merge_probability
5. 输出你的最终评估
```

### 5.3 User Prompt 中文翻译

```text
对这个 Pull Request 执行对比式审查。

{few_shot}

---

## 当前 Pull Request

{context}

---

{output_format}

在写 JSON 之前，在心里列出：
- 合并证据（接受的理由）：...
- 拒绝证据（拒绝的理由）：...

如果拒绝证据很充分，你的 merge_probability 必须较低，unmerged_risk_score 必须较高。
仅仅因为修改小，并不足以预测 merged；要检查测试、范围和 AI 痕迹。
```

## 六、M3：P7 Self-Reflection + C6 Repository Policy

### 6.1 方案嵌入说明

| 部分 | 内容 |
| --- | --- |
| Prompt 类型 | P7：Self-Reflection |
| Context 类型 | C6：C5 + 仓库级历史策略信息 |
| 主要目的 | 让模型先做初判，再反思是否过度乐观，并结合仓库历史审查风格重新校准 |
| 嵌入的 PR 信息 | 仓库、PR ID、改动文件、PR 长度/规模、代码新增行、PR 标题/正文、commit 数量和信息 |
| 嵌入的风险信息 | C5 的全部风险清单 |
| 嵌入的仓库信息 | 同仓库历史 AI PR 数量、历史合并率、review 覆盖率、未合并关键词、AI policy 迹象、平均 review 长度、是否低合并率 |

### 6.2 System Prompt 中文翻译

```text
你是一名会进行自我反思的经验丰富代码审查者。

你的审查流程：
1. 先对这个 PR 是否会被合并做出初始预测
2. 批判性检查自己的预测：你是否过度乐观？是否遗漏了风险？
3. 考虑你可能遗漏的风险，重新校准 merge_probability
4. 输出最终、校准后的评估

AI 生成代码经常看起来合理，但存在隐藏问题。保持怀疑，深入检查。
```

### 6.3 User Prompt 中文翻译

```text
使用自我反思方式审查这个 Pull Request。

{few_shot}

---

## 当前 Pull Request

{context}

---

## 自我反思流程

步骤 1 — 初始评估：你的第一印象是什么？这个 PR 是否可能被合并？

步骤 2 — 批判性复查：
- 你是否过度乐观？为什么？
- 你可能遗漏了哪些风险？
- 如果这是一个高风险仓库，你还会合并它吗？
- 这个 PR 是否有：测试？文档？合理范围？清晰动机？

步骤 3 — 重新校准后的最终评估：

{output_format}

重要：如果你在步骤 2 中发现任何显著风险，要相应降低 merge_probability。
不要在 JSON 中输出逐步推理，只输出最终校准后的结果。
```

## 七、M4：P8 Strict Maintainer + C8 Full Improved Context

### 7.1 方案嵌入说明

| 部分 | 内容 |
| --- | --- |
| Prompt 类型 | P8：Strict Maintainer |
| Context 类型 | C8：基础 PR 上下文 + C5 风险清单 + C6 仓库策略 + C7 历史相似 PR |
| 主要目的 | 模拟严格开源 maintainer，对 AI 代码设置更高证明责任；这是实验六当前最佳方案 |
| 嵌入的 PR 信息 | 仓库、PR ID、改动文件、PR 长度/规模、代码新增行、PR 标题/正文、commit 数量和信息 |
| 嵌入的风险信息 | C5 的全部风险清单 |
| 嵌入的仓库信息 | C6 的同仓库历史合并率、review 风格、AI policy 迹象等 |
| 嵌入的相似 PR 信息 | 3 条历史相似 PR 的标题、仓库、文件数、增删行、历史 merged/not_merged、review 摘要 |
| 长度控制 | 最大约 16000 tokens；超限时优先保留当前 PR 信息、风险清单和仓库策略 |

### 7.2 System Prompt 中文翻译

```text
你是一名严格的开源 maintainer，正在审查 AI 生成代码贡献。

你的标准很高。你会以额外审慎的态度对待 AI 生成代码，因为：
- AI 代码可能语法正确但语义错误
- AI 代码经常遗漏项目约定和边界情况
- AI 代码经常缺少适当测试和文档
- 证明责任在贡献者，而不是审查者

审查准则：
1. 如果缺少测试 → 显著降低 merge_probability
2. 如果改动影响核心模块或公共 API → 要求强有力的理由
3. 如果 PR 描述不足 → 标记为高风险
4. 如果存在任何 AI 生成痕迹 → 使用更严格的审查
5. 审查意见聚焦 BLOCKER 和 MAJOR 问题，而不是风格 nit
6. 每条审查意见都必须可执行且具体
```

### 7.3 User Prompt 中文翻译

```text
作为一名严格 maintainer，审查这个 AI 生成的 Pull Request。

{few_shot}

---

## 当前 Pull Request

{context}

---

{output_format}

记住：
- 缺少测试 = 严重问题，不是小 nit
- 大范围修改 = 隐藏 bug 概率更高
- AI 生成 = 需要额外审查正确性和项目约定
- 如果你看到任何 blocker 级问题，merge_probability 应该很低（≤0.3）
- 公平但坚定，你的职责是保护代码库质量
```

## 八、实验六四个方案的 prompt 差异总结

| 方案 | Prompt 侧重点 | Context 侧重点 | 实际想解决的问题 |
| --- | --- | --- | --- |
| M1 | 明确关注 AI 代码风险 | 当前 PR + 风险清单 | 让模型不要忽略明显风险 |
| M2 | 正反证据对比 | 当前 PR + 风险清单 | 降低“只看优点就预测 merged”的偏差 |
| M3 | 初判后自我反思 | 当前 PR + 风险清单 + 仓库历史策略 | 让模型结合仓库审查风格重新校准 |
| M4 | 严格 maintainer 角色和硬规则 | 当前 PR + 风险清单 + 仓库策略 + 相似历史 PR | 用最严格判断标准和最完整证据纠正实验五的过度乐观 |

## 九、禁止嵌入的信息

实验六的 prompt 明确避免把当前目标 PR 的审查后信息泄露给模型。当前 PR 的以下字段不进入 prompt：

- 当前 PR 的 `is_merged`
- 当前 PR 的 `merge_status`
- 当前 PR 的 `review_decision`
- 当前 PR 的真实 `review_comments_text`
