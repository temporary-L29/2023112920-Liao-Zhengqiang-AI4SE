# 实验六计划书：基于实验五结果改进 AI 生成代码 PR 的代码审查

## 0. 给执行者的总说明

本计划书面向从 0 开始接手的 Claude Code Agent。执行者需要先理解整个系列实验的目标，再在实验五最新结果基础上补充实验六，而不是重新设计一套孤立实验。

本系列实验的统一应用场景是：

> 在 Pull Request 刚刚提交时，根据提交当时已经可获得的信息，预测该 PR 是否可能被 merge，并生成初步代码审查意见。

因此，任何实验中的 prompt、上下文、特征都必须模拟“PR 刚提交”的真实场景。不能把审查之后才出现的信息放进目标 PR 输入。

实验六的核心任务不是重新采集数据，也不是重新训练模型，而是在实验五已经完成的 AI 生成代码 PR 评估基础上，通过 **Prompt 优化** 和 **上下文增强** 改进 DeepSeek 对 AI 代码 PR 的审查能力，并与实验四、实验五结果做可比对比。

执行架构：

- **Claude Code Agent**：工程执行者，负责编写/修改代码、构建任务、调用 API、解析输出、评估、画图、写报告。
- **DeepSeek API**：被调用的大语言模型后端，只负责对每个 PR 输出预测和代码审查意见。
- DeepSeek 不是工程 agent，不要把它写成执行实验的主体。

实验三未实现，因此实验六中所有涉及 Code2Vec / CodeBERT 的内容均标记为 `N/A`，不补做、不伪造。

## 1. 指导书要求与实验六定位

根据指导书中实验六的要求，本实验应体现以下内容：

1. 面向 AI 生成代码 PR，改进已有代码审查方法。
2. 与前面实验结果比较，尤其是实验四 DeepSeek 方法和实验五 AI 代码迁移结果。
3. 比较不同 Prompt 和不同上下文组合的效果，而不是只跑单个方法。
4. 同时关注 merge prediction 和 review comment generation。
5. 分析改进方法是否真正提升了代码审查价值，而不是只提高 Accuracy 或 F1。

结合实验五最新结果，实验六的具体定位是：

> 实验五已经完成 AI 代码上的原始 LLM 4×4 基线。实验六要在同一批 AI PR 上运行改进 Prompt × 增强 Context 矩阵，并证明改进方法是否能降低实验五中“默认预测 merged”的过度乐观偏差。

实验六必须回答三个问题：

1. 改进 prompt/context 是否比实验五原始 `B01-B16` 基线更能识别 unmerged AI PR？
2. 哪类改进最有效：风险清单、仓库策略、历史相似 PR、还是严格 maintainer 角色？
3. 改进是否以可接受的 Recall/F1 代价换来了更高 Specificity、Balanced Accuracy 和更低 False Positive Rate？

## 2. 信息可用性约束

### 2.1 可以进入目标 PR prompt 的信息

这些信息在 PR 刚提交时已经可获得，允许使用：

- PR title
- PR body
- commit messages
- changed files list
- diff / patch
- additions / deletions / changed files count
- 文件类型统计
- 是否包含测试文件
- 从 diff 即时计算的复杂度、AST/CFG 摘要
- 仓库名、主要语言类型
- 历史 PR 统计或历史示例，但必须来自目标 PR 之前或训练/验证集合，不能泄漏当前目标 PR 结果

### 2.2 禁止进入目标 PR prompt 的信息

这些是审查后或最终结果信息，不能作为目标 PR 输入：

- `is_merged`
- `merge_status`
- `review_decision`
- `num_reviewers`
- `num_review_comments`
- `num_inline_comments`
- `review_comments_text`
- `approved`
- `lgtm`
- `changes requested`
- 当前 PR 的真实 review comment
- 当前 PR 的 reviewer 数量或评论数量
- `has_ai_generated_code` 作为模型输入特征

这些字段只能用于：

- 构建 AI 数据集筛选条件。
- 计算评价指标。
- BLEU/ROUGE reference。
- 报告分析。

## 4. 实验六总体方案

实验六在实验五基础上补充如下实验：

1. 复用实验五 `ai_llm_sample_50.csv`。
2. 对同一批 50 条 AI PR 构建增强上下文 C5-C8。
3. 运行改进 prompt P5-P8 与增强上下文 C5-C8 的 4×4 矩阵，共 16 组。
4. 将实验六 `I01-I16` 与实验五 `B01-B16` 放在同一张表、同一批样本上比较。
5. 选择最佳改进组合，在 Balanced-120 / Hard-FP-40 上验证稳定性。
6. 如果预算允许，对最佳 1 个组合跑 Full-343 复验。

注意：

- 实验五 `B01-B16` 已完成，不需要重复调用。
- 实验六必须生成 `I01-I16`，不能只跑 M1-M4 四组。
- 现有 M1-M4 结果若已经存在，只能作为验证结果，不是主实验矩阵的替代。

## 6. 上下文设计

### 6.1 实验五原始上下文 C1-C4

这些上下文已经在实验五使用，实验六只用于解释对比，不需要重复构建：

| Context | 名称 | 内容 |
| --- | --- | --- |
| C1 | Diff Only | changed files + diff / patch + additions / deletions |
| C2 | Diff + PR Description | C1 + PR title + PR body |
| C3 | Diff + PR Description + Commit | C2 + commit messages |
| C4 | Full Early Context | C3 + AST/CFG/复杂度摘要 + 修改规模 + 文件类型统计 |

### 6.2 实验六增强上下文 C5-C8

实验六新增 4 种上下文，全部必须满足早期信息可用性约束。

#### C5 Risk Checklist Context

目标：显式列出 AI 生成代码 PR 的风险检查项，帮助模型不要默认预测 merged。

包含：

- 是否缺少测试。
- 是否修改核心模块。
- 是否大范围改动。
- PR 描述是否不足。
- 是否只做格式化或机械批量改动。
- 是否涉及 API / backward compatibility。
- 是否涉及安全、权限、并发、资源释放。
- 是否出现 AI 工具痕迹，如 Copilot / ChatGPT / Claude / Cursor / Codex，但只能作为风险提示，不作为标签输入。

输出为结构化摘要，不泄漏真实 merge 结果。

#### C6 Repository Policy Context

目标：加入仓库级审查偏好和风险规则。

来源：

- 实验五数据集中非目标 PR 的统计。
- 公开仓库贡献指南或可从历史数据计算的规则摘要。
- 不使用目标 PR 的 review 结果。

包含：

- 该仓库 AI PR 大致合并率。
- 该仓库常见拒绝原因摘要。
- 是否要求测试、文档、issue link。
- 对大范围改动、生成代码、自动化 PR 的可能态度。

注意：

- C6 不能直接告诉模型“目标 PR 是否 merged”。
- 历史统计应以仓库级粗粒度方式呈现。

#### C7 Historical Similar PR Context

目标：提供历史相似 PR 的早期信息模式，模拟 maintainer 参考历史经验。

检索规则：

- 从训练/历史 PR 或实验五非目标样本中检索。
- 不允许包含当前目标 PR。
- 相似度基于 title/body/changed files/diff summary。
- 每个目标 PR 最多提供 3 条历史相似 PR。

历史示例中允许包含：

- 历史 PR 的 title/body 摘要。
- 历史 PR 的 changed files summary。
- 历史 PR 的风险摘要。

历史示例中不建议直接展示：

- 历史 PR 的完整 review comment。
- 历史 PR 的过长 diff。

如果为了 few-shot 风格必须展示历史 outcome，只能用于历史样本，不能来自 test target；并且报告中要说明这是历史监督信号，不是目标 PR 泄漏。

#### C8 Full Improved Context

目标：组合 C5 + C6 + C7，并加入当前 PR 的 C4 级别早期信息。

包含：

- 当前 PR title/body/commit/diff summary。
- 当前 PR 文件与规模摘要。
- 当前 PR AST/CFG/复杂度摘要。
- C5 风险清单。
- C6 仓库策略。
- C7 历史相似 PR。

限制：

- 控制 prompt 总长度，建议最大 12k-16k tokens。
- diff 过长时保留文件摘要、关键 hunks、测试相关 hunks、API 修改 hunks。

## 7. Prompt 设计

### 7.1 实验五原始 Prompt P1-P4

这些 prompt 已在实验五基线使用：

| Prompt | 名称 | 说明 |
| --- | --- | --- |
| P1 | Zero-shot | 直接判断 merge probability 并生成 review comment |
| P2 | Few-shot | 加入人类 PR validation 示例 |
| P3 | Role-based | 设定为开源项目 maintainer |
| P4 | Chain-of-Thought Style | 要求结构化分析，但最终只输出简短依据 |

实验六不重复调用 P1-P4，直接读取实验五 `llm_4x4_metrics.json`。

### 7.2 实验六改进 Prompt P5-P8

#### P5 AI-Risk-Aware Prompt

目标：显式关注 AI 生成代码的审查风险。

要求模型输出：

- `risk_factors`
- `missing_tests`
- `scope_risk`
- `compatibility_risk`
- `maintainability_risk`
- `unmerged_risk_score`
- merge prediction
- review comments

#### P6 Contrastive Prompt

目标：强制模型同时列出支持 merge 和反对 merge 的证据，避免单向乐观。

要求模型输出：

- `merge_evidence`
- `reject_evidence`
- `risk_balance_summary`
- calibrated merge probability
- review comments

#### P7 Self-Reflection Prompt

目标：让模型先做初判，再检查是否过度乐观。

要求：

- 初步判断。
- 检查缺测试、大范围、描述不足、仓库策略冲突。
- 给出校准后的最终预测。
- 只输出短 `reflection_summary`，不要输出长推理。

#### P8 Strict Maintainer Prompt

目标：模拟严格 maintainer，对 AI 生成代码采用更高审查标准。

规则：

- 缺少测试且修改非文档代码时，降低 merge probability。
- 修改核心模块、大范围文件、兼容性风险时，必须考虑 `not_merged`。
- PR 描述不足或 commit 信息弱时，降低 confidence。
- 评论优先指出 blocker/major 问题。

## 11. 评估标准与成功条件

实验六不能只用 Accuracy / F1 判断是否成功。实验五已经证明高 F1 可能只是因为模型倾向预测 `merged`。

### 11.1 主指标

优先级从高到低：

1. Specificity 提升：能识别更多真实 unmerged PR。
2. FPR 下降：减少把 unmerged 错判为 merged。
3. Balanced Accuracy 提升：同时考虑 merged 与 unmerged。
4. ROC-AUC / PR-AUC 提升：概率排序更有区分度。
5. F1 不发生灾难性下降。

### 11.2 与实验五基线比较

实验六最佳组合必须至少与以下基线比较：

| 基线 | 说明 |
| --- | --- |
| B14 | 实验五 Balanced-50 最佳 F1，F1=0.7059 |
| B04 | 实验五 Balanced-50 最佳 AUC，AUC=0.6352 |
| B12 | 实验五 Balanced-50 最佳 Specificity，Specificity=0.36 |
| P2_C3 full-343 | 实验五低成本 full-343 代表组合 |
| P2_C4 full-343 | 实验五低成本 full-343 代表组合 |

### 11.3 成功阈值

实验六方法可以接受 Recall/F1 小幅下降，但必须带来更真实的风险识别能力。

建议成功标准：

1. Specificity 相比 B14 至少提升 0.20。
2. FPR 相比 B14 至少下降 0.20。
3. Balanced Accuracy 高于 B14。
4. ROC-AUC 不低于 B14，或至少高于 0.60。
5. F1 不低于 B14 的 85%。

如果 I16 或其他组合 F1 低于 B14，但 Specificity 和 Balanced Accuracy 明显更高，应解释为：模型从“默认合并”变为“风险敏感”，这是符合实验六目的的改进。
