# 实验四修订计划书：基于大语言模型的人类编写代码代码审查

## 1. 修订原因

指导书明确要求实验四完成两个并列任务：

1. Merge Prediction：预测 PR 是否可能被 merge。
2. Review Comment Generation：调用大语言模型生成代码审查意见。

此前计划把两者合并在同一次 DeepSeek 调用和同一个 JSON 输出中，虽然工程实现已经包含 `review_comments`、BLEU/ROUGE 评价和生成样例，但计划表述更偏向 Merge Prediction，容易造成“忽略代码审查意见生成任务”的问题。因此本修订版将 Review Comment Generation 单独列为核心实验任务。

## 2. 总体目标

本实验模拟 **Pull Request 刚提交时** 的真实应用场景：根据 PR 提交时可获得的信息，调用 DeepSeek 大语言模型同时完成：

1. 预测该 PR 是否有可能被 merge。
2. 生成初步代码审查意见，帮助开发者尽早发现风险。

实验三未单独实现，因此实验四复用实验二的 test split 和数据处理产物，作为实验三测试数据准备的替代。

## 3. 信息可用性约束

可以进入 prompt 的信息：

- PR title
- PR body
- commit messages
- changed files list
- diff / patch
- 修改文件数、新增行数、删除行数
- 从 diff 计算出的 AST / CFG / 复杂度摘要
- 仓库名、文件类型、主要语言

不能进入目标 PR prompt 的信息：

- `is_merged`
- `merge_status`
- `review_decision`
- `num_reviewers`
- `num_review_comments`
- `num_inline_comments`
- `review_comments_text`
- `approved`、`lgtm`、`changes requested` 等审查后标签

真实 `review_comments_text` 只能用于 Review Comment Generation 的评价 reference，不能作为目标 PR 输入。

## 4. 数据与调用规模

数据来源：

- `experiment2/results/processed/splits.csv`
- `experiment2/results/processed/patch_index.json`
- `experiment2/results/processed/features_main.csv`
- `experiment2/results/processed/ast_features.csv`
- `experiment2/results/processed/cfg_features.csv`

抽样策略：

- 从实验二 `test` split 固定抽取 50 条人类代码 PR。
- 每个仓库 10 条。
- 尽量保持 merged / unmerged 比例接近测试集分布。
- 优先保证至少 25 条有真实 `review_comments_text`，用于 Review Comment Generation 评价。

调用矩阵：

```text
4 种 Prompt × 4 种 Context × 50 条 PR = 800 次 DeepSeek API 调用
```

每次调用同时输出 Merge Prediction 和 Review Comment Generation 结果，避免成本翻倍。

## 5. Prompt 与 Context

Prompt 类型：

| ID | 名称 | 说明 |
| --- | --- | --- |
| P1 | Zero-shot | 直接要求模型预测合并可能性并生成审查意见 |
| P2 | Few-shot | 加入 4 个 validation split 示例 |
| P3 | Role-based | 设定为大型开源项目 senior maintainer |
| P4 | Chain-of-Thought Style | 要求结构化分析，但最终只输出简短 JSON |

Context 类型：

| ID | 名称 | 说明 |
| --- | --- | --- |
| C1 | Diff Only | 文件路径 + diff / patch |
| C2 | Diff + PR Description | C1 + PR title + PR body |
| C3 | Diff + PR Description + Commit | C2 + commit messages |
| C4 | Full Early Context | C3 + AST/CFG 摘要 + 规模统计 + 文件类型 |

## 6. 统一输出格式

所有 prompt 必须要求 DeepSeek 输出 JSON：

```json
{
  "merge_prediction": "merged 或 not_merged",
  "merge_probability": 0.0,
  "confidence": "low/medium/high",
  "evidence_summary": "一句话说明预测依据",
  "review_comments": [
    {
      "file": "path or unknown",
      "line": null,
      "severity": "nit/minor/major/blocker",
      "comment": "review text"
    }
  ]
}
```

其中 `review_comments` 是 Review Comment Generation 任务的核心输出，不是附属字段。

## 7. Merge Prediction 评价

对 16 个 Prompt × Context 组合分别计算：

- Accuracy
- Precision
- Recall
- F1-score
- ROC-AUC
- Confusion Matrix
- JSON parse fail rate
- 平均推理时间

并与实验二 main 模型对比：

- SVM main
- Random Forest main

不能与实验二 upper bound 直接对比，因为 upper bound 使用了审查过程信息。

## 8. Review Comment Generation 评价

这是实验四必须单独展示的第二个任务。

评价范围：

- 只在真实 `review_comments_text` 非空的样本上计算 BLEU/ROUGE。
- 对所有样本统计生成评论数量和 severity 分布。

指标：

- BLEU
- ROUGE-1
- ROUGE-2
- ROUGE-L
- 平均生成评论数
- `nit/minor/major/blocker` severity 分布
- 无真实评论样本是否仍生成评论

输出文件：

```text
results/evaluation/comment_generation_metrics.json
results/evaluation/best_generated_examples.json
figures/06_bleu_rouge_comparison.png
```

报告必须展示：

1. Review Comment Generation 指标表。
2. BLEU/ROUGE 对比图。
3. 至少 3 条自动生成的代码审查意见样例。
4. 对 BLEU/ROUGE 偏低的解释：代码审查意见表达多样，低 n-gram 重合不必然代表质量差。

## 10. 报告结构修订

实验报告必须明确分成两个任务：

1. Merge Prediction 实验结果
   - 16 组指标表
   - heatmap
   - 与实验二传统 ML 对比

2. Review Comment Generation 实验结果
   - BLEU / ROUGE 指标表
   - 生成评论数量与 severity 分布
   - 自动生成审查意见样例
   - 自动指标局限性分析

这样才能完整覆盖指导书 4.6、4.7、4.8 中关于“调用大语言模型生成代码审查意见”的要求。

## 11. 对后续实验的影响

实验五会复用实验四的两类能力：

1. Merge Prediction：在 AI 生成代码 PR 上测试合并预测泛化能力。
2. Review Comment Generation：在 AI 生成代码 PR 上测试自动审查意见质量。

因此实验四的 `prompt_templates.py`、`context_builder.py`、`api_caller.py` 和 `evaluator.py` 必须保留双任务输出结构，不能只保留分类预测。
