# 实验二：代码审查意见挖掘与需求理解 — 计划书

## 一、实验概述

实验二在实验一构建的 1500 条 GitHub Pull Request 数据集基础上，采用**传统机器学习方法**训练 **Merge Prediction（PR 是否合并）** 分类模型。核心任务包括：数据筛选（仅保留人类编写代码的 PR）、代码中间表示（AST/CFG）生成、多维度特征工程、SVM 与 Random Forest 模型训练与评估。

本实验产出的模型将作为后续深度学习实验（实验三~七）的**性能基线（Baseline）**。

## 四、各步骤详细计划

### 步骤一：数据筛选与清洗 (`data_filter.py`)

**输入：** `experiments/experiment1/results/processed/dataset.csv`

**处理逻辑：**

1. **AI 代码过滤**：过滤 `has_ai_generated_code == True` 的记录（约 103 条，占 6.87%）
   - 注：`has_ai_reviewer` 的 PR 暂时保留，因为 AI reviewer 不影响代码本身是人类编写的判断
2. **合并状态过滤**：仅保留 `merge_status` 为 `"merged"` 或 `"closed_not_merged"` 的记录，去除 `"open"` 状态
3. **缺失值处理**：
   - 数值字段：`num_changed_files` 为 0 的标记为异常（PR 不可能改 0 个文件）→ 剔除
   - 文本字段：`title`、`body` 为空的用 `"<EMPTY>"` 填充
4. **文本规范化**：对 `title`、`body`、`commit_messages` 做基本清洗（去除多余空白、控制字符）
5. **输出统计**：生成筛选后的数据摘要（各仓库分布、合并率、字段分布）

**输出：**
- `results/processed/human_only_dataset.csv`
- 打印筛选统计日志到 `results/pipeline.log`

**关键注意事项：**
- 步骤一不改变原始 csv 文件，全部写入 experiment2 目录

---

### 步骤二：代码中间表示生成

#### 2a. AST 提取 (`ast_extractor.py`)

**输入：** 实验一的 `merged_raw.json`（需要其中的 `files[].patch`），仅处理步骤一筛选后保留的 PR

**技术方案：**
- 使用 `tree-sitter` 库（官方 Python binding）
- 根据文件扩展名自动选择语言 parser：
  | 扩展名 | 语言 |
  |--------|------|
  | `.py` | Python |
  | `.js` `.jsx` `.ts` `.tsx` | JavaScript / TypeScript |
  | `.go` | Go |
  | `.java` | Java |
  | `.c` `.cpp` `.h` `.hpp` `.cc` | C / C++ |
  | `.rs` | Rust |
  | 其他 / 无 patch | 跳过，标记 `ast_available=False` |

**提取策略：**
- 从 patch 中提取**所有添加行**（+ 开头），拼接为代码片段
- 对拼接后的代码片段进行 AST 解析（注：patch 片段可能不完整，需做好 try/except 容错）
- 统计指标：
  - `ast_total_nodes`：AST 节点总数（所有文件汇总）
  - `ast_max_depth`：AST 最大深度
  - `ast_node_types`：各节点类型数量分布（dict → 后续展平为特征）
  - `ast_files_parsed`：成功解析的文件数
  - `ast_parse_success_rate`：成功解析文件数 / 总变更文件数

**容错策略：**
- 单个文件解析失败不影响其他文件
- 整个 PR AST 全部失败时，AST 特征置为 0 或 -1（标记缺失），不剔除该样本

#### 2b. CFG 构建 (`cfg_extractor.py`)

**技术方案：**
- 基于 AST 节点，识别控制流相关节点类型，构建简化 CFG
- 使用 `networkx` 存储图结构（DiGraph）
- 控制流节点类型（以 Python/JS 为例）：
  - `if_statement` → 分支节点
  - `for_statement` / `while_statement` → 循环节点
  - `try_statement` / `except_clause` → 异常处理节点
  - `return_statement` → 终止节点
  - `function_definition` / `class_definition` → 函数/类入口节点
  - `break_statement` / `continue_statement` → 跳转节点

**CFG 简化规则：**
- 代码片段 CFG 与完整文件 CFG 不同，patch 片段缺少上下文
- 策略：按添加行出现的顺序逐步构建基本块，识别分支/循环关键字后连接边
- 统计指标：
  - `cfg_total_nodes`：CFG 节点总数
  - `cfg_total_edges`：CFG 边总数
  - `cfg_cyclomatic_complexity`：圈复杂度（edges - nodes + 2）
  - `cfg_max_branch_depth`：最大分支嵌套深度

**容错策略：**
- 无 AST → 无 CFG，标记 `cfg_available=False`
- CFG 构建失败的 PR 不剔除，CFG 特征置为 0

**重要说明：** 步骤二是本实验**技术实现难度最高的部分**。AST/CFG 的提取质量取决于 patch 片段的完整性和 tree-sitter 语言的覆盖率。计划预期在 5 种语言（Python/JavaScript/TypeScript/Go/Java）上获取较高质量的 AST，其余语言（Ruby/C++/Rust）可能覆盖率有限。

---

### 步骤三：特征提取 (`feature_extractor.py`)

**输入：** 步骤一的筛选数据集 + 步骤二的 AST/CFG 特征

**特征分类（共约 25-35 维）：**

#### A. 代码修改特征（来自实验一原始字段）
| 特征名 | 含义 | 来源字段 |
|--------|------|----------|
| `num_changed_files` | 修改文件数 | dataset 直接使用 |
| `total_additions` | 新增行数 | dataset 直接使用 |
| `total_deletions` | 删除行数 | dataset 直接使用 |
| `num_commits` | 提交数量 | dataset 直接使用 |
| `code_churn` | 代码扰动 | additions + deletions（派生） |
| `additions_per_file` | 每文件新增行数 | additions / changed_files（派生） |
| `deletions_per_file` | 每文件删除行数 | deletions / changed_files（派生） |

#### B. 文本特征
| 特征名 | 含义 | 来源 |
|--------|------|------|
| `title_len` | PR 标题长度（字符数） | title |
| `body_len` | PR 描述长度 | body |
| `pr_length` | title + body 总长度 | dataset 直接使用 |
| `commit_msg_len` | Commit Message 总长度 | commit_messages |
| `num_commits` | Commit 数量（与 A 类重复，取一处） | num_commits |

#### C. AST 结构特征
| 特征名 | 含义 |
|--------|------|
| `ast_total_nodes` | AST 节点总数（所有文件汇总） |
| `ast_max_depth` | AST 最大嵌套深度 |
| `ast_files_parsed` | 成功解析的文件数 |
| `ast_parse_success_rate` | 解析成功率 |
| `ast_if_count` | if 语句数量 |
| `ast_for_count` | for 循环数量 |
| `ast_while_count` | while 循环数量 |
| `ast_func_def_count` | 函数定义数量 |
| `ast_class_def_count` | 类定义数量 |
| `ast_try_count` | try 语句数量 |
| `ast_return_count` | return 语句数量 |
| `ast_assignment_count` | 赋值语句数量 |

#### D. CFG 特征
| 特征名 | 含义 |
|--------|------|
| `cfg_total_nodes` | CFG 节点总数 |
| `cfg_total_edges` | CFG 边总数 |
| `cfg_cyclomatic_complexity` | 圈复杂度 |
| `cfg_max_branch_depth` | 最大分支嵌套深度 |
| `cfg_entry_nodes` | 入口节点数（≈函数数） |
| `cfg_exit_nodes` | 出口节点数 |
| `cfg_avg_out_degree` | 平均出度 |

#### E. 审查特征（来自实验一）
| 特征名 | 含义 |
|--------|------|
| `num_reviewers` | 审阅者人数 |
| `num_review_comments` | 评论数 |
| `num_inline_comments` | 行内评论数 |
| `num_labels` | 标签数 |
| `review_intensity` | 审查强度 = (review_comments + inline_comments) / changed_files（派生） |

**数据预处理：**
1. 对所有数值特征做 `StandardScaler` 标准化（训练集 fit，测试集 transform）
2. 缺失值：AST/CFG 特征缺失用 0 填充（并新增 `is_ast_missing` 二值标记）
3. 处理异常值：用 IQR 法或百分位截断（如 `total_additions > 99th_percentile` 截断为 99th）
4. 类别特征编码：`repo` 做 One-Hot Encoding（5维）

**输出：**
- `results/processed/features.csv`：特征矩阵（~1400行 × ~35列）
- `results/processed/features_stats.txt`：特征描述性统计（均值/方差/min/max/缺失率）

### 步骤四：模型训练 (`model_trainer.py`)

**输入：** `features.csv`

**数据集划分：**
- 训练集 70% / 测试集 30%，分层抽样（`stratify=y` 保证合并/未合并比例一致）
- 随机种子：`random_state=42`

**模型一：SVM（Support Vector Machine）**
- 核函数：RBF（默认）、Linear（备选）
- 超参数调优（GridSearchCV + 5-fold CV）：
  - `C`: [0.1, 1, 10, 100]
  - `gamma`: ['scale', 'auto', 0.01, 0.1]
  - `kernel`: ['rbf', 'linear']
- 类别权重：`class_weight='balanced'`（处理数据不平衡）

**模型二：Random Forest**
- 超参数调优（GridSearchCV + 5-fold CV）：
  - `n_estimators`: [100, 200, 500]
  - `max_depth`: [10, 20, 30, None]
  - `min_samples_split`: [2, 5, 10]
  - `min_samples_leaf`: [1, 2, 4]
- 类别权重：`class_weight='balanced'`

**输出：**
- `results/models/svm_model.pkl`
- `results/models/rf_model.pkl`
- 训练日志（最佳超参数、训练时间、CV 分数）

## 六、关键设计决策与说明

### 6.1 为什么不使用深度学习？

这是实验设计的有意为之——实验二使用传统 ML（SVM + RandomForest）建立基线，实验三及后续实验才会引入深度学习模型。这样做的意义：
- 验证手工特征 + 传统模型能否解决 Merge Prediction 任务
- 为后续深度学习模型提供性能下界（baseline）
- 通过 RF 特征重要性分析，理解哪些特征对代码审查最有预测力

### 6.2 AST/CFG 提取的局限性

- patch 片段是局部代码，不是完整文件，AST 解析可能不完整
- 多语言支持需要多个 tree-sitter 语言包，安装和维护成本较高
- 如果某语言的 parser 不可用，对应文件的 AST 特征将缺失
- **计划中优先支持 Python、JavaScript、TypeScript、Go**（覆盖 5 个目标仓库的主要语言），Java/C++/Rust 作为扩展

### 6.3 数据不平衡处理

从实验一可知合并率约 63%，不算极端不平衡。但仍需：
- `class_weight='balanced'`
- 分层抽样保证训练/测试集分布一致
- 除 Accuracy 外重点关注 F1 和 ROC-AUC

### 6.4 与实验一的衔接

- 实验二**直接读取**实验一的 `dataset.csv` 和 `merged_raw.json`，不重新采集数据
- 实验二筛选掉 `has_ai_generated_code==True` 的 PR（~103条），剩余 ~1397 条
- 对 `has_ai_reviewer==True` 的 PR 暂不剔除（AI 审查不影响代码是人类编写的判断）
