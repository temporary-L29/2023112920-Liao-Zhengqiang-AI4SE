# 实验五计划书：针对大模型生成代码的代码审查

## 0. 总体定位

实验五的目标是分析已有代码审查方法在 **AI 生成代码 Pull Request** 上的泛化能力。根据指导书，原本应比较实验二、实验三、实验四构建的模型；但本项目当前没有完成实验三，因此本实验采用以下口径：

1. 实验二传统机器学习模型：保留并测试。
2. 实验三 Code2Vec / CodeBERT：跳过，并在报告中明确说明未实现，不能伪造结果。
3. 实验四 DeepSeek 大语言模型方法：保留并测试。

本实验不是重新训练模型，而是做 **跨场景泛化测试**：已有模型在人类代码数据上建立，现在直接用于 AI 生成代码 PR，观察 Merge Prediction 和 Review Comment Generation 的性能变化。

整个系列实验的真实应用场景是：**在 Pull Request 刚刚提交时，尽早预测该 PR 是否可能被 merge，并给出初步代码审查意见**。因此实验五仍然必须遵守信息可用性约束：

- 可以使用：PR title、PR body、commit messages、changed files、diff / patch、新增删除行数、修改文件数、从 diff 计算出的 AST/CFG 摘要、仓库名、语言、文件类型。
- 不能作为当前目标 PR 输入：`is_merged`、`merge_status`、`review_decision`、`num_reviewers`、`num_review_comments`、`num_inline_comments`、`review_comments_text`、`approved/lgtm/changes requested` 等审查后标签。
- `has_ai_generated_code` 只用于筛选实验五数据集，不能作为模型输入特征。
- 真实 review comments 只能用于 Review Comment Generation 的 reference 和误差分析。

执行架构：

- **Claude Code Agent**：负责实现工程流程、数据采集、模型调用、评估、图表和报告。
- **DeepSeek API**：作为实验五中的 LLM 后端，被 Claude Code Agent 调用。
- DeepSeek 不是工程实现 agent，不负责写代码或调度实验。

# 第一部分：AI 生成代码数据采集与数据集构建

## 1.1 目标

第一部分只解决一个问题：构建一个可用于实验五的 AI 生成代码 PR 数据集。

数据集需要满足：

1. PR 为 closed 状态，能确定最终是否 merged。
2. 尽量包含 PR title、body、commit messages、changed files、diff / patch。
3. 尽量包含真实 review comments，供生成任务评价。
4. 标记 `has_ai_generated_code=True`，并记录 `ai_detection_reason`。
5. 不把 AI 标签、review 信息、merge 信息作为后续模型输入。

## 1.2 当前 seed 数据事实

实验一已有 AI 生成代码 PR，可作为 seed 数据：

| 指标 | 数值 |
| --- | ---: |
| 总 PR 数 | 1500 |
| AI 生成代码 PR | 97 |
| AI 生成代码合并率 | 82.47% |
| 有真实 review comment 的 AI PR | 88 |

按仓库分布：

| 仓库 | AI PR 数 | Merged | Unmerged | 真实评论非空 |
| --- | ---: | ---: | ---: | ---: |
| facebook/react | 2 | 0 | 2 | 0 |
| huggingface/transformers | 2 | 1 | 1 | 1 |
| kubernetes/kubernetes | 1 | 0 | 1 | 0 |
| microsoft/vscode | 88 | 77 | 11 | 87 |
| pandas-dev/pandas | 4 | 2 | 2 | 0 |

直接使用 seed 数据可以跑通基础实验，但存在三个明显问题：

1. 样本只有 97 条，规模偏小。
2. 88 条来自 VSCode，仓库分布极不均衡。
3. 合并率 82.47%，类别不均衡，Accuracy 容易虚高。

因此，实验五建议把数据构建分为两个版本：

- **Seed 数据集**：实验一已有 97 条，作为保底方案。
- **Enhanced 数据集**：重新拉取/扩充 AI 生成代码 PR，作为推荐主方案。

## 1.3 数据采集策略

### 1.3.1 Seed 数据构建

输入：

- `experiment1/results/processed/dataset.csv`
- `raw/merged_raw.json`

处理：

1. 筛选 `has_ai_generated_code == True`。
2. 保留全部 seed 样本，不做 train/val/test 切分。
3. 从 `raw/merged_raw.json` 抽取 patch、files、commits、reviews、review comments。
4. 生成 seed 数据统计。

输出：

```text
experiment5/results/processed/ai_generated_dataset_seed.csv
experiment5/results/processed/ai_patch_index_seed.json
experiment5/results/evaluation/seed_dataset_stats.json
```

### 1.3.2 Enhanced 数据重新拉取

如果有 GitHub Token，执行增强采集。目标是把 AI 生成代码 PR 扩充到 200-300 条，提高实验可信度。
尽量保证仓库分布均衡，不要按时间顺序拉取，因为一些仓库在特定时间段有集群效应，在条件约束下随机拉取
推荐采集顺序：

1. 原始 5 个仓库优先：
   - `microsoft/vscode`
   - `huggingface/transformers`
   - `facebook/react`
   - `pandas-dev/pandas`
   - `kubernetes/kubernetes`
2. 如果样本不足，再加入扩展仓库：
   - `vercel/next.js`
   - `microsoft/playwright`
   - `langchain-ai/langchain`
   - `openai/openai-python`
   - `microsoft/autogen`

注意：新增仓库会降低与实验二人类代码基线的可比性。因此报告必须分别展示：

- 原始 5 仓库子集结果。
- 扩展仓库全集结果。

### 1.3.3 AI 生成代码识别规则

增强采集采用关键词规则，不做主观人工标注。关键词必须写入配置，保证可复现。

强信号关键词：

- `copilot`
- `github copilot`
- `chatgpt`
- `gpt`
- `claude`
- `cursor`
- `codex`
- `code agent`
- `ai-generated`
- `generated by ai`
- `generated with`
- `co-authored-by: copilot`
- `openai`
- `deepseek`
- `qwen`

匹配字段：

- PR title
- PR body
- commit messages
- labels
- author / bot name
- review comments

说明：

- review comments 可以用于判断 PR 是否涉及 AI 工具，也可作为后续生成任务 reference。
- review comments 不能进入 Merge Prediction 输入。

筛选规则：

1. 至少命中 1 个强信号关键词，才进入候选集。
2. 如果只命中弱词 `ai`，必须同时命中 `generated`、`agent`、`copilot`、`llm` 等上下文词。
3. 每条样本必须记录 `ai_detection_reason`，例如：`body contains "generated by copilot"`。
4. 每条增强样本保留原始 API 响应，便于复核。

### 1.3.4 采集字段

增强采集字段尽量与实验一一致：

- PR 基本信息：`pr_id, pr_number, repo, title, body, author, created_at`
- 合并信息：`is_merged, merge_status`
- 代码修改：`files, patch, additions, deletions, changed_files_list`
- commit 信息：`commit_messages`
- review 信息：`review_decision, review_comments_text`
- 标签：`label_names`
- AI 判断：`has_ai_generated_code=True, ai_detection_reason`

其中合并信息和 review 信息只能用于评价和分析，不能进入模型输入。

### 1.3.5 采集规模与成本控制

建议：

1. 总增强样本目标 200-300 条。
2. 每个仓库最多保留 50-80 条，避免再次被单一仓库主导。
3. 只采集 `state=closed` 的 PR。
4. 优先使用 GitHub Search API 查询候选 PR，再对候选 PR 拉取详情、files、commits、reviews、comments。
5. 支持断点续跑，保存 candidates 和 collected raw data。
6. 如果 GitHub Token 不可用或 API 限额不足，自动回退到 seed 数据集。

## 1.4 数据去重与版本选择

去重规则：

- 以 `repo + pr_number` 作为唯一键。
- 实验一 seed 样本优先保留。
- 新采集样本如果字段更完整，可以补充 patch/review，但不能覆盖原始 ground truth。

最终输出两个可用版本：

```text
ai_generated_dataset_seed.csv       # 只含实验一 97 条
ai_generated_dataset_enhanced.csv   # seed + 新采集样本
ai_generated_dataset.csv            # 当前实验实际使用版本
```

版本选择规则：

1. 如果 enhanced 样本数 >= 150，默认 `ai_generated_dataset.csv` 指向 enhanced。
2. 如果 enhanced 样本数 < 150 或采集失败，默认使用 seed。
3. 报告中必须说明最终使用哪个版本，以及原因。

## 2.6 DeepSeek 大语言模型实验

实验五的核心是测试实验四大语言模型方法在 AI 代码上的泛化能力。当前实验五已经完成两层 LLM 评估：

1. full-343 低成本迁移测试：`P2_C3` / `P2_C4`。
2. Balanced-50 全组合基线：`P1-P4 × C1-C4 = B01-B16`。

因此，后续实验六应直接复用实验五 4×4 基线结果，而不是再只拿 `P2_C3` / `P2_C4` 两个点比较。

### 2.6.1 低成本默认组合

| 组合 | 说明 | 目的 |
| --- | --- | --- |
| P2_C3 | Few-shot + Diff + PR Description + Commit | 使用实验四人类代码上 F1 较好的组合 |
| P2_C4 | Few-shot + Full Early Context | 测试 AST/CFG/结构摘要对 AI 代码是否有帮助 |

调用规模：

- Seed 数据：约 `97 × 2 = 194` 次 DeepSeek API 调用。
- Enhanced 数据：若 200-300 条，则约 `400-600` 次调用。

可选扩展：

- `P1_C3`：观察 zero-shot 是否更稳。
- `P3_C3`：观察 role-based prompt 是否更适合 AI 代码。

### 2.6.2 已完成：AI 代码上的 Prompt × Context 全组合基线

为了让实验五完整承接实验四，并为实验六提供充分基线，已完成：

```text
4 Prompt × 4 Context × 50 AI PR = 800 次 DeepSeek API 调用
```

组合：

| 组合 ID | Prompt | Context | 说明 |
| --- | --- | --- | --- |
| B01 | P1 Zero-shot | C1 Diff Only | 只看 diff 的零样本判断 |

| B03 | P1 Zero-shot | C3 Diff + PR Description + Commit | 加入 commit messages |
| B04 | P1 Zero-shot | C4 Full Early Context | 加入结构和规模摘要 |
| B05 | P2 Few-shot | C1 Diff Only | Few-shot + 纯 diff |

| B08 | P2 Few-shot | C4 Full Early Context | 对齐已有 `P2_C4`，但在 50 条样本上重算 |
| B09 | P3 Role-based | C1 Diff Only | 维护者角色 + 纯 diff |
| B10 | P3 Role-based | C2 Diff + PR Description | 维护者角色 + PR 描述 |

数据集：

## 2.7 评估与对比

`evaluator.py` 输入：

- `traditional_ml_predictions.csv`
- `llm_parsed_predictions.csv`
- `experiment2/results/evaluation/metrics_main.json`
- `experiment4/results/evaluation/metrics_by_prompt_context.json`
- `experiment4/results/evaluation/comment_generation_metrics.json`

处理：

1. 计算 SVM / Random Forest 在 AI PR 上的指标。
2. 计算 DeepSeek 在 AI PR 上的指标。
3. 对比人类代码与 AI 代码：
   - 实验二 SVM / Random Forest human test 指标 vs AI test 指标。
   - 实验四 P2_C3 human sample 指标 vs AI test 指标。
   - 实验四 P2_C4 human sample 指标 vs AI test 指标。
4. 对真实评论非空样本计算 Review Comment Generation 指标：
   - BLEU
   - ROUGE-1
   - ROUGE-2
   - ROUGE-L
5. 做错误分析：
   - False Positive：预测会合并但实际未合并。
   - False Negative：预测不会合并但实际合并。
   - 按仓库、修改规模、是否测试文件、评论数量分组统计。

输出：

```text
results/evaluation/traditional_ml_metrics.json
results/evaluation/llm_metrics.json
results/evaluation/human_vs_ai_comparison.json
results/evaluation/comment_generation_metrics.json
results/evaluation/error_analysis.json
```

注意：

- 如果使用 enhanced 数据，必须同时输出原始 5 仓库子集和扩展全集的指标。
- 如果 seed 数据仍由 VSCode 主导，报告必须强调仓库偏置。
- ROC-AUC 在 unmerged 样本太少时可能不稳定，要结合混淆矩阵解释。

## 2.10 对后续实验的影响

实验五的结论会直接指导实验六：

1. 如果传统 ML 性能下降明显，说明人工特征对 AI 代码风格变化不够鲁棒，实验六应强化 LLM 上下文增强。
2. 如果 DeepSeek 对 AI PR 过度乐观，实验六应设计更严格的风险识别 prompt。
3. 如果 P2_C4 优于 P2_C3，说明 AST/CFG 和结构摘要对 AI 代码有帮助。
4. 如果 BLEU/ROUGE 很低，不一定说明评论无价值，因为代码审查意见表达多样；实验六可增加人工样例分析或规则化评价。
5. 如果 enhanced 数据暴露更多 unmerged AI PR，实验六应优先使用 enhanced 数据做 prompt 改进。

## 2.13 预期结论方向

1. AI PR 合并率高，所有模型可能倾向于预测 merged，因此 Accuracy 可能看起来不低。
2. 由于模型明显偏向 `merged`，Specificity、FPR、Balanced Accuracy 和混淆矩阵比单独 Accuracy/F1 更重要。
3. 传统 ML 可能受仓库分布影响较大，尤其 seed 数据主要来自 VSCode。
4. DeepSeek 可能生成更完整的审查意见，但 Merge Prediction 可能偏乐观。
5. 4×4 基线中 `B14=P4_C2` 的 F1 最好，`B04=P1_C4` 的 AUC 最好，`B12=P3_C4` 的 Specificity 最好；实验六应分别与这些基线比较。
6. BLEU/ROUGE 可能偏低，原因是代码审查意见表达多样，不能只靠文本重合度判断质量。
7. 实验五最重要的价值是为实验六指出改进方向：AI 代码风险检查清单、上下文增强和更严格 prompt。
