# 实验六相对实验五性能提升原因分析

## 1. 文档目的

本文说明实验六为什么比实验五取得更好的 AI 生成代码 PR 审查效果，重点分析项目中实际实现的上下文增强（C5-C8）和 Prompt 优化（P5-P8），并结合在同一批 50 条 AI PR 上运行的 B01-B16 与 I01-I16 主实验结果解释性能变化。

本实验的应用场景保持不变：在 PR 刚提交时，利用当时可获得的信息预测 PR 是否可能被 merge，并生成初步代码审查意见。

实验六不是重新训练 DeepSeek，而是在实验五相同 AI PR 数据和同一批 Balanced-50 样本上，改变模型看到的信息组织方式和判断规则。

---

## 2. 实验五存在的主要问题

### 2.1 模型严重偏向预测 merged

实验五在 Balanced-50 上的最佳 F1 组合是 B14（P4 Chain-of-Thought + C2 Diff/PR Description）：

| 指标 | B14 |
| --- | ---: |
| Accuracy | 0.6000 |
| Precision | 0.5581 |
| Recall | 0.9600 |
| F1 | 0.7059 |
| ROC-AUC | 0.5880 |
| Specificity | 0.2400 |
| FPR | 0.7600 |
| 混淆矩阵 | TN=6, FP=19, FN=1, TP=24 |

B14 在 50 条 PR 中将 43 条预测为 merged，只预测 7 条 not_merged。高 Recall 和较高 F1 主要来自“尽量预测 merged”，但 25 条真实 unmerged 中只识别出 6 条。

实验五各基线组合的平均 merge probability 大多处于 0.72-0.88，说明这种乐观倾向不是某一个 Context 独有，而是原始 Prompt 的共同问题。

### 2.2 原始 Prompt 没有强制寻找拒绝理由

实验五 P1-P4 的主要变化是 zero-shot、few-shot、角色设定和结构化分析，但最终仍然是“根据 PR 信息判断是否合并”。Prompt 没有明确要求：

- 不要默认预测 merged。
- 主动寻找缺测试、范围过大、破坏兼容性等拒绝信号。
- 同时输出 merged 与 not_merged 两侧证据。
- 当存在 blocker 时强制降低 merge probability。
- 对模型自己的乐观判断进行校准。

因此 DeepSeek 容易把“描述完整、代码看起来合理、改动较小”直接解释为可合并。

### 2.3 上下文只描述当前 PR，缺少决策环境

实验五 C1-C4 主要包含当前 PR 的 diff、title/body、commit 和结构摘要，没有显式提供：

- 当前 PR 是否缺少测试。
- 是否涉及核心模块或 public API。
- 是否是大范围修改。
- 仓库对 AI PR 的历史接受程度。
- 同类历史 PR 的结果。
- 仓库常见的拒绝模式。

但是实验五数据表明仓库差异很大：VS Code AI PR 合并率约 90%，React 约 34.6%。仅看代码本身无法解释这种社区和仓库政策差异。

## 3. 实验六的上下文修改

实验六在 PR 提交时可获得的 title、body、commit、文件列表、diff/patch 与变更规模信息基础上，构造 C5-C8。新增信息的目标不是加入审查后结果，而是把已有风险信号和历史决策环境组织成模型可直接使用的证据。

### 3.1 C5：AI Code Risk Checklist

C5 在基础上下文后增加自动风险检查清单，将原本隐藏在 diff 和文件列表中的信息转换成模型容易使用的显式信号。

实际实现检查八类风险：

1. `MISSING_TESTS`：没有测试文件。
2. `LARGE_SCOPE`：修改文件数大于 10。
3. `LARGE_ADDITIONS`：新增行数大于 500。
4. `CONFIG_DEP_CHANGE`：修改配置、CI、依赖或 lock 文件。
5. `CORE_MODULE`：文件路径可能涉及 core、API、interface、export 等关键区域。
6. `SHORT_DESCRIPTION`：PR body 小于 100 字符。
7. `AI_TOOL_TRACES`：title/body 中出现 Copilot、GPT、Claude、Cursor、Codex 等痕迹。
8. `API_CHANGE_NO_DOCS`：可能修改 public/export API，但没有文档文件变化。

C5 的作用不是直接给出标签，而是把审查流程从“先看优点”改为“先做风险扫描”。

仅使用 C5 时，P5+C5（I01）的 Specificity 已达到 0.400，相比 B14 的 0.240 提升 0.160；说明显式风险清单本身能够减少一部分乐观误判。

### 3.2 C6：Repository Policy Context

C6 = 基础上下文 + C5 风险清单 + 仓库历史策略信息。

实际实现加入：

- 当前仓库除目标 PR 外的 AI PR 数量。
- 历史 AI PR 合并率。
- 历史 AI PR 的评论覆盖率。
- 历史 review comment 平均长度。
- unmerged AI PR 标题中的常见关键词。
- 历史评论中是否存在 AI policy 讨论。
- 当仓库 AI PR 合并率低于 50% 时，明确提示该仓库审查更严格。

C6 解决的是实验五缺失的“决策环境”问题。同样的代码规模在 VS Code 和 React 中可能对应不同的合并概率，仓库历史信息能帮助模型进行概率校准。

结果上，C6 的四个组合平均表现为：

| 指标 | C6 平均值 |
| --- | ---: |
| F1 | 0.7141 |
| ROC-AUC | 0.6577 |
| Specificity | 0.4933 |

I14（P8 Strict Maintainer + C6）成为综合最佳组合，说明“严格判断标准 + 仓库政策”比单纯增加更多文本更有效。

### 3.3 C7：Historical Similar PR Context

C7 = 基础上下文 + C5 风险清单 + 3 条历史相似 PR。

相似 PR 按以下因素打分：

- 相同仓库优先，权重最高。
- 修改文件数接近。
- 新增行数接近。
- title 关键词重叠。

每条历史 PR 展示 title、仓库、修改规模、merged/not_merged outcome 和 review summary。

C7 的主要作用是提供可类比案例，让模型不再只依赖抽象规则，而是参考“类似 PR 过去发生了什么”。

结果上，C7 四个组合的平均 ROC-AUC 为 0.7089，是四种增强 Context 中最高的。I07（P6+C7）的 ROC-AUC 达到 0.7358，为全部 32 组中的最高值。这说明历史相似 PR 对概率排序特别有帮助，但不一定能直接得到最佳分类阈值。

### 3.4 C8：Full Improved Context

C8 将以下内容全部组合：

- 当前 PR 的早期信息与变更内容。
- C5 风险清单。
- C6 仓库政策。
- C7 历史相似 PR。

C8 设置 16000 token 上限，超长时优先保留当前 PR 信息、风险清单和仓库策略。

C8 四个组合的平均 Specificity 为 0.5162，是四种 Context 中最高的。P8+C8（I16）取得最高 Specificity=0.6667 和最低 FPR=0.3333。

但 C8 不是所有指标都最好。I14（P8+C6）的 F1=0.7547，高于 I16 的 0.6800。可能原因是 C8 信息更长，历史样例和仓库规则同时存在时会稀释模型对当前 PR 的关注；C6 更短、更聚焦，因而综合效果更好。

## 4. 实验六的 Prompt 修改

### 4.1 统一输出从“预测”升级为“风险校准”

实验六输出格式新增：

- `unmerged_risk_score`
- `risk_factors`
- 更明确的 `confidence`
- 带文件、严重级别和可执行建议的 `review_comments`

统一 Prompt 明确要求：

- 不得默认预测 merged。
- 主动寻找 missing tests、large scope、breaking change、AI artifact。
- 评论必须指向具体文件、函数或行为。
- 没有风险时也要解释为什么。

这些约束让模型必须显式建模“不合并风险”，而不是只给一个乐观概率。

### 4.2 Few-shot 示例改为平衡的正反案例

实验六提供 2 个 merged 和 2 个 not_merged 示例，且示例不把“是否使用 AI”直接等同于质量：

- 小范围、测试充分的 Copilot 类型标注可以 merged。
- 大范围、无测试的 GPT 重构应 not_merged。
- breaking API 且无文档的 Claude 辅助实现应 not_merged。

模型由此学习到：AI 痕迹只是风险因素，测试、范围、兼容性和文档才是决定性证据。

### 4.3 P5：AI-Risk-Aware

P5 将角色设定为专门审查 AI 代码的 reviewer，要求关注：

- hallucinated API。
- 缺少边界条件。
- 过度设计。
- 测试、文档、兼容性和维护性。

P5 四个 Context 的平均 F1=0.7139，为 P5-P8 中最高；平均 Specificity=0.4746。它在保持 Recall 的同时，能温和地降低 merged 偏差。

### 4.4 P6：Contrastive

P6 要求模型先分别列出支持 merge 与反对 merge 的证据，再进行权衡；当证据接近时降低 merge probability。

P6 与 C7 组合时 AUC 最好，说明正反证据比较有助于概率排序。但 P6 平均 Specificity 只有 0.3942，模型即使识别到了拒绝证据，最终分类仍然偏乐观。

### 4.5 P7：Self-Reflection

P7 要求模型完成：初步判断、检查自己是否过度乐观、寻找遗漏风险、重新校准概率。

P7 的平均 Specificity=0.3839，没有明显优于 P6。实际结果说明，仅要求模型“反思”不一定足以改变最终决策；如果缺少硬性判定规则，模型反思后仍可能回到 merged 默认值。

### 4.6 P8：Strict Maintainer

P8 是性能提升最明显的 Prompt。它不只要求分析风险，而是设置了明确的决策规则：

- 缺少测试时显著降低 merge probability。
- 修改 core/public API 时要求更强依据。
- 描述不足时判为高风险。
- AI 痕迹出现时提高审查强度。
- 存在 blocker 时 merge probability 不应高于 0.3。
- 评论重点放在 blocker/major，而不是 style nit。

P8 四个组合的平均结果：

| 指标 | P8 平均值 |
| --- | ---: |
| F1 | 0.6822 |
| ROC-AUC | 0.6664 |
| Specificity | 0.5812 |
| FPR | 0.4188 |
| 平均 merge probability | 0.5974 |

其他 Prompt 的平均 Specificity 分别为 P5=0.4746、P6=0.3942、P7=0.3839。P8 的硬规则比“平衡分析”或“自我反思”更能改变模型的默认决策边界。

## 5. 性能为什么上涨

### 5.1 I14：综合性能提升原因

I14 = P8 Strict Maintainer + C6 Repository Policy。

| 指标 | 实验五 B14 | 实验六 I14 | 变化 |
| --- | ---: | ---: | ---: |
| Accuracy | 0.6000 | 0.7234 | +0.1234 |
| Precision | 0.5581 | 0.7143 | +0.1562 |
| Recall | 0.9600 | 0.8000 | -0.1600 |
| F1 | 0.7059 | 0.7547 | +0.0488 |
| ROC-AUC | 0.5880 | 0.6927 | +0.1047 |
| Specificity | 0.2400 | 0.6364 | +0.3964 |
| Balanced Accuracy | 0.6000 | 0.7182 | +0.1182 |
| FPR | 0.7600 | 0.3636 | -0.3964 |

I14 的平均 merge probability 从 B14 的 0.8104 降到 0.6096。预测 merged 的数量从 B14 的 43/50 降到 I14 的 28/47 个有效结果。

性能上涨的直接链条是：

1. C5 把缺测试、范围和 API 风险显式化。
2. C6 提供仓库接受 AI PR 的历史背景。
3. P8 用硬规则将风险转化为较低 merge probability。
4. False Positive 从 19 降到 8，Precision、Specificity 和 Balanced Accuracy 同时提升。

### 5.2 I16：unmerged 识别提升原因

I16 = P8 Strict Maintainer + C8 Full Improved Context。

I16 的 Specificity=0.6667、FPR=0.3333，混淆矩阵为 TN=16、FP=8、FN=8、TP=17，是 unmerged 识别最强的组合。

其平均 merge probability 进一步降到 0.5663，49 个有效结果中预测 25 个 merged、24 个 not_merged，输出分布已经接近平衡。

I16 的代价是 Recall 降到 0.68，说明完整上下文和严格 Prompt 会把一部分真实 merged PR 也判为高风险。因此：

- 需要综合准确性时，选择 I14。
- 更重视发现不会合并的 PR 时，选择 I16。

### 5.3 Review Comment Generation 的变化

P8 不仅改变分类结果，也改变了评论风格：

| 指标 | I14 | I16 |
| --- | ---: | ---: |
| Risk Coverage | 1.0000 | 0.9773 |
| Actionability | 0.9296 | 0.9275 |
| 评论具体性 | 0.3901 | 0.4864 |
| Blocker+Major 比例 | 41.8% | 51.0% |
| 平均评论数/PR | 4.6 | 5.4 |

这与 P8 的实现一致：评论不再集中于格式或轻微建议，而是更多指出测试、兼容性、范围和维护风险。

## 6. Prompt 与 Context 的相对贡献

实验结果表明，Prompt 是改变分类阈值的主要驱动，Context 是提供证据和校准背景的放大器。

### Prompt 的作用

- P8 将平均 merge probability 降到 0.5974。
- P8 平均 Specificity 比 P7 高约 0.197。
- 硬规则比一般性的“请反思”更能改变 merged 默认偏置。

### Context 的作用

- C5 提供最低成本的风险显式化。
- C6 最适合综合分类，I14 的 F1 和 Balanced Accuracy 最好。
- C7 最适合概率排序，平均 AUC 最高。
- C8 最适合最大化 unmerged 识别，I16 的 Specificity 最高。

因此不能简单说“上下文越多越好”。上下文必须与 Prompt 配合：P8 能把 C6/C8 中的风险信息转化为更严格的判断，而 P6/P7 即使看到同样信息，最终仍较偏向 merged。

## 7. 必须说明的局限性

性能提升是真实测得的，但并不能全部解释为模型代码理解能力提升。

### 7.1 C6/C7 存在历史信息时间边界问题

C6 和 C7虽然排除了目标 PR 自身，但候选历史数据来自完整 343 条数据集，没有按 `created_at < target.created_at` 过滤，也没有排除同一评估集中的其他 PR。

C6 使用了其他 PR 的 merge outcome 和 review 统计；C7直接展示相似 PR 的 outcome 与 review summary。这不是目标标签直接泄漏，但可能使用了目标 PR 提交时尚未发生的“未来历史”，会放大 I07、I14、I16 的效果。

真实部署中必须：

- 只使用目标 PR 提交时间之前已经关闭的历史 PR。
- 不允许从当前 test/evaluation 集互相检索 outcome。
- 仓库 merge rate 只能用历史 train/validation 数据计算。

### 7.2 C5 输入了 AI 标记

C5 会在 `has_ai_generated_code=True` 时告诉模型该 PR 被标记为 AI 生成。所有实验五样本都属于 AI 数据，因此它不是区分 merged/unmerged 的标签，但会整体提高模型的严格程度。

如果真实系统在 PR 刚提交时不能可靠判断代码是否由 AI 生成，就不应使用该字段，应只保留 title/body/commit 中可观察到的 AI 痕迹。

### 7.3 样本规模和解析率

主实验每组只有 50 条，且 I14 只有 47 条成功解析、I16 有 49 条成功解析。不同组合的有效样本不完全相同，目前没有 bootstrap 置信区间或显著性检验，因此小幅指标差异不应被解释得过强。

## 8. 答辩讲解建议

可以按以下逻辑讲解：

> 实验五的问题不是 Recall 不够，而是模型过度乐观。以 B14 为例，Recall 达到 0.96，但 Specificity 只有 0.24，25 个不会合并的 PR 中有 19 个被误判。实验六从 Context 和 Prompt 两侧同时改进。Context 侧加入风险清单、仓库政策和历史相似 PR；Prompt 侧要求模型显式建模 unmerged risk，并用 Strict Maintainer 的硬规则降低缺测试、大范围和 API 风险 PR 的合并概率。最终 I14 将 F1 从 0.706 提升到 0.755，Specificity 从 0.24 提升到 0.636，FPR 从 0.76 降到 0.364；I16 的 Specificity 进一步达到 0.667。结果说明性能提升的关键不是让模型获得审查后信息，而是将提交时可见的风险证据和历史决策环境转化为更严格的判断标准。

随后应主动补充：

> 目前 C6/C7 的历史数据没有严格按时间切分，可能带来一定乐观估计。后续应只使用目标 PR 提交前的历史数据，并在更大样本上做显著性检验。

这种讲法既能说明实验六的实际贡献，也能清楚交代实验边界。

---

## 9. 最终结论

实验六相对实验五的性能提升来自三个层次：

1. **风险信息显式化**：C5 将测试、范围、API、文档和 AI 痕迹转成结构化风险清单。
2. **决策环境补充**：C6/C7 引入仓库政策和历史相似 PR，使模型能根据社区差异校准概率。
3. **判断规则重设**：P8 用严格 maintainer 角色和硬规则打破“默认 merged”的决策习惯。

其中，P8 是降低 False Positive 的主要驱动，C6 提供最佳综合性能，C7 提供最佳概率排序，C8 配合 P8 提供最强的 unmerged 识别能力。

由于历史数据尚未严格按时间隔离，最终报告将结果描述为“Prompt/Context 增强下的实验性能提升”，不直接声称全部来自模型对 AI 代码的本质理解提升。
