# 实验二计划书：基于机器学习的人类编写代码代码审查

## 1. 实验定位

实验二承接实验一构建的 GitHub Pull Request 数据集，目标是使用传统机器学习方法完成 Merge Prediction，即预测一个 Pull Request 是否最终会被合并。按照指导书要求，本实验必须完成：

1. 从实验一数据集中筛选人类编写代码对应的 PR。
2. 生成代码中间表示，包括 AST 和 CFG。
3. 提取代码结构特征、代码修改特征、文本特征和统计特征。
4. 训练 SVM 与 Random Forest 分类模型。
5. 用 Accuracy、Precision、Recall、F1-score、ROC-AUC 等指标评估模型。
6. 分析 Random Forest 特征重要性，并形成实验报告。

本实验的战略作用不只是完成一次分类任务，而是为实验三至实验七提供传统机器学习基线。因此结果必须“可复现、可解释、无明显信息泄漏”。如果为了提高分数直接使用审查结果、approved/lgtm 标签等后验信息，模型指标可能很好看，但会破坏后续深度学习、大语言模型和 AI 生成代码实验的对比意义。

## 3. 核心设计原则

### 3.1 主实验避免信息泄漏

Merge Prediction 的主实验应模拟“PR 提交后、审查结论出来前”的预测场景。因此主模型只使用提交时或代码修改本身可获得的信息：

- PR 标题、正文、Commit Message 的长度和文本质量统计。
- 修改文件数、新增行、删除行、Commit 数、文件类型分布。
- patch 中新增代码的 AST/CFG 结构特征。
- 仓库 One-Hot 编码，用于控制项目差异。

以下字段不进入主实验模型：

- `review_decision`
- `num_reviewers`
- `num_review_comments`
- `num_inline_comments`
- `review_comments_text`
- `label_names` 中的 `approved`、`lgtm`、`changes requested` 等后验标签

原因：这些信息通常发生在代码审查过程中或审查之后，直接用于预测合并会导致数据泄漏。可以在扩展实验中加入这些字段，作为“含审查过程信息的性能上界”，但不能作为最终主结论。

### 3.2 保留 AI Reviewer，不保留 AI Generated Code

指导书要求筛选“人类编写代码”对应的 PR，因此：

- 删除：`has_ai_generated_code == True` 的 97 条 PR。
- 保留：`has_ai_reviewer == True` 的 PR。

理由：AI Reviewer 说明审查者可能是 AI，但不代表代码本身由 AI 生成。保留这些样本可以维持数据量，并且对后续实验五“AI 生成代码”形成清晰边界。

### 3.3 建立统一数据划分，服务后续实验

实验二应输出一个稳定的划分文件，后续实验三、四可以复用同一测试集，从而保证传统 ML、深度学习、LLM 的结果可横向比较。

建议划分：

- Train：70%
- Validation：15%
- Test：15%

划分方式：

- 使用 `random_state=42`
- 按 `repo + is_merged` 分层
- 输出 `results/processed/splits.csv`，至少包含 `pr_id, repo, is_merged, split`

实验二训练时：

- SVM / Random Forest 的超参数选择只使用 train 和 validation。
- 最终报告指标只在 test 上汇报。

## 6. 模块级实施计划

### 6.1 `data_filter.py`

输入：

- `experiment1/results/processed/dataset.csv`

处理：

1. 读取 CSV，统一布尔字段格式。
2. 保留 `has_ai_generated_code == False`。
3. 保留 `merge_status` 为 `merged` 或 `closed_not_merged` 的样本。
4. 删除明显异常样本：
   - `num_changed_files <= 0`
   - `total_additions < 0`
   - `total_deletions < 0`
5. 文本缺失值填充为空字符串：
   - `title`
   - `body`
   - `commit_messages`
6. 保留 `has_ai_reviewer` 作为分析字段，但默认不进入主特征。
7. 输出筛选统计，包括每个仓库的人类代码数量、合并率、AI Reviewer 占比。

输出：

- `results/processed/human_only_dataset.csv`
- `results/processed/human_dataset_stats.json`

验收标准：

- 输出约 1403 条样本；如果剔除异常样本后少量减少，需要在日志中说明。
- 每个仓库都仍有 merged 和 unmerged 两类样本。

### 6.2 `split_dataset.py`

输入：

- `human_only_dataset.csv`

处理：

1. 构造分层键：`repo + "_" + is_merged`。
2. 用 `train_test_split` 两次完成 70/15/15 划分。
3. 如果某个分层桶数量太小，退化为只按 `is_merged` 分层，并记录警告。

输出：

- `results/processed/splits.csv`

验收标准：

- Train、Validation、Test 中每个仓库都尽量出现。
- 各 split 的合并率接近整体人类代码合并率。

### 6.3 `patch_indexer.py`

输入：

- `raw/merged_raw.json`
- `human_only_dataset.csv`

处理：

1. 按 `detail.id` 与 `pr_id` 对齐。
2. 为每个 PR 抽取：
   - 文件名
   - 扩展名
   - additions / deletions / changes
   - patch
   - status
3. 从 patch 中提取新增代码行：
   - 保留以 `+` 开头的代码行。
   - 排除 `+++ b/file` 这类 diff header。
   - 保留缩进，因为缩进对 Python AST/CFG 有用。
4. 标记文件类型：
   - code：`.py`, `.go`, `.js`, `.jsx`, `.ts`, `.tsx`, `.java`, `.c`, `.cpp`, `.h`, `.hpp`, `.rs`
   - config/data：`.json`, `.yml`, `.yaml`, `.toml`, `.xml`
   - doc：`.md`, `.rst`, `.txt`
   - other

输出：

- `results/processed/patch_index.json`
- `results/processed/patch_coverage_stats.json`

验收标准：

- PR 级 patch 覆盖率应接近 99%。
- 文件级 patch 覆盖率应接近 96%。
- 没有 patch 的 PR 不删除，只标记 `patch_available=False`。

### 6.4 `ast_extractor.py`

输入：

- `patch_index.json`

优先技术方案：

- 使用 `tree-sitter` 或可用的 tree-sitter language package。
- 优先支持 `.py`, `.go`, `.js`, `.jsx`, `.ts`, `.tsx`。
- P1 支持 `.rs`, `.java`, `.c`, `.cpp`。

降级方案：

如果 tree-sitter 安装困难或某语言 parser 不可用，必须实现 lexical AST fallback。fallback 不生成完整语法树，而是基于新增代码行构建轻量结构树：

- 缩进层级或大括号层级作为深度。
- `def/function/func/class/interface/struct` 作为定义节点。
- `if/else/switch/case/for/while/try/catch/return` 作为语法节点。
- 普通语句作为 statement 节点。

这样即使环境缺少 parser，也能满足“构建 AST 表示并统计 AST 节点”的实验要求，并且报告中可说明这是 patch 片段上的轻量 AST。

提取特征：

| 特征 | 说明 |
| --- | --- |
| `ast_total_nodes` | AST 总节点数 |
| `ast_max_depth` | AST 最大深度 |
| `ast_files_parsed` | 成功解析文件数 |
| `ast_files_fallback` | 使用 fallback 的文件数 |
| `ast_parse_success_rate` | 成功解析率 |
| `ast_error_node_count` | tree-sitter ERROR 节点数 |
| `ast_func_def_count` | 函数定义数量 |
| `ast_class_def_count` | 类/结构定义数量 |
| `ast_if_count` | if/switch/case 数量 |
| `ast_loop_count` | for/while/range 循环数量 |
| `ast_try_count` | try/catch/except 数量 |
| `ast_return_count` | return/yield 数量 |
| `ast_assignment_count` | 赋值语句数量 |

输出：

- `results/processed/ast_features.csv`
- `figures/03_ast_cfg_coverage.png` 的 AST 覆盖数据

验收标准：

- 所有 PR 都有一行 AST 特征。
- 解析失败不能中断流水线。
- 至少输出 2 个 AST 样例到报告中，可以选择 Python 和 TypeScript/Go 各一个。

### 6.5 `cfg_extractor.py`

输入：

- `patch_index.json`
- `ast_features.csv`

推荐实现：

构建 patch 级简化 CFG，不追求完整编译器级控制流。原因是 PR patch 只有局部新增行，缺少完整上下文，强行构建精确 CFG 成本高且容易失败。简化 CFG 更可行，也更适合课程实验。

CFG 节点规则：

- `entry`：每个可解析代码文件一个入口节点。
- `statement`：连续普通语句合并为基本块。
- `branch`：`if`, `else`, `elif`, `switch`, `case`。
- `loop`：`for`, `while`, `range`。
- `exception`：`try`, `catch`, `except`, `finally`。
- `exit`：`return`, `raise`, `throw`, `break`, `continue`。

CFG 边规则：

- 顺序语句建立顺序边。
- 分支节点至少建立 true/false 两类边。
- 循环节点建立进入边和回边。
- return/raise/throw 指向 exit。

提取特征：

| 特征 | 说明 |
| --- | --- |
| `cfg_total_nodes` | CFG 节点数 |
| `cfg_total_edges` | CFG 边数 |
| `cfg_branch_nodes` | 分支节点数 |
| `cfg_loop_nodes` | 循环节点数 |
| `cfg_exit_nodes` | 退出节点数 |
| `cfg_cyclomatic_complexity` | `E - N + 2P`，P 为连通分量近似 |
| `cfg_max_branch_depth` | 最大分支嵌套深度 |
| `cfg_avg_out_degree` | 平均出度 |
| `cfg_available` | 是否成功构建 |

输出：

- `results/processed/cfg_features.csv`

验收标准：

- 所有 PR 都有一行 CFG 特征。
- 至少输出 1 个 CFG 样例图或文本结构到实验报告。
- 如果 CFG 是近似构建，报告中必须明确说明原因：patch 片段缺少完整函数和文件上下文。

### 6.6 `feature_extractor.py`

输入：

- `human_only_dataset.csv`
- `splits.csv`
- `ast_features.csv`
- `cfg_features.csv`

输出两套特征矩阵。

#### A. 主实验特征：`features_main.csv`

主实验只使用无明显后验泄漏的特征。

代码修改特征：

- `num_changed_files`
- `total_additions`
- `total_deletions`
- `num_commits`
- `code_churn = total_additions + total_deletions`
- `net_lines = total_additions - total_deletions`
- `additions_per_file`
- `deletions_per_file`
- `commits_per_file`
- `large_pr_flag`：是否超过人类数据中 code_churn 的 75 分位数

文本特征：

- `title_len`
- `body_len`
- `commit_msg_len`
- `title_word_count`
- `body_word_count`
- `commit_msg_word_count`
- `has_issue_link`
- `has_release_note`
- `body_empty_flag`

文件类型特征：

- `num_code_files`
- `num_doc_files`
- `num_config_files`
- `num_test_files`
- `test_file_ratio`
- `code_file_ratio`
- 主要语言 One-Hot 或计数：`has_py`, `has_go`, `has_js`, `has_ts`, `has_rs`

结构特征：

- 全部 AST 特征。
- 全部 CFG 特征。

仓库控制特征：

- `repo` One-Hot。

标签：

- 目标列：`is_merged`
- 辅助列：`pr_id`, `repo`, `split`

#### B. 上界扩展特征：`features_upper_bound.csv`

在主实验特征基础上加入审查过程特征：

- `num_reviewers`
- `num_review_comments`
- `num_inline_comments`
- `num_labels`
- `review_decision` One-Hot
- `has_ai_reviewer`
- 可选安全标签：只加入非明显后验的标签数量统计，不直接使用 `approved/lgtm`

这套特征只用于讨论“如果审查过程信息可见，模型性能上限如何”，不能作为主实验结论。

预处理：

1. 数值特征用 train 集 fit `StandardScaler`。
2. 缺失值用 0 填充，并保留缺失标记字段。
3. 极端值用 99 分位截断，防止 3000 文件这种异常大 PR 主导 SVM。
4. 类别变量使用 One-Hot。
5. 不使用 `author`，因为作者身份容易造成记忆化，不利于后续泛化。

验收标准：

- `features_main.csv` 至少 35 个有效特征。
- `features_upper_bound.csv` 独立存在，但报告中明确标记为扩展对照。

### 6.7 `model_trainer.py`

输入：

- `features_main.csv`
- `features_upper_bound.csv`

必须训练：

1. SVM
2. Random Forest

建议额外训练：

1. `DummyClassifier(strategy="most_frequent")`
2. `LogisticRegression(class_weight="balanced")`

额外模型不写入指导书主标题，但作为 sanity check 很有价值。如果 SVM/RF 没有明显超过 Dummy，说明特征或划分存在问题。

SVM 参数：

```python
{
    "clf__kernel": ["linear", "rbf"],
    "clf__C": [0.1, 1, 10],
    "clf__gamma": ["scale", 0.01, 0.1]
}
```

Random Forest 参数：

```python
{
    "clf__n_estimators": [200, 500],
    "clf__max_depth": [8, 16, None],
    "clf__min_samples_split": [2, 5, 10],
    "clf__min_samples_leaf": [1, 2, 4],
    "clf__max_features": ["sqrt", "log2"]
}
```

训练策略：

- 使用 train 训练。
- 使用 validation 选择最佳超参数。
- 最终只在 test 上评估一次。
- `class_weight="balanced"`。
- 固定随机种子 42。
- 模型保存时同时保存 scaler/encoder，推荐使用 sklearn `Pipeline`。

输出：

- `results/models/svm_main.joblib`
- `results/models/rf_main.joblib`
- `results/models/svm_upper_bound.joblib`
- `results/models/rf_upper_bound.joblib`
- `results/evaluation/training_log.json`

### 6.8 `model_evaluator.py`

测试集指标：

- Accuracy
- Precision
- Recall
- F1-score
- ROC-AUC
- PR-AUC，作为补充指标
- Confusion Matrix

必须输出：

1. 主实验 SVM vs Random Forest 指标表。
2. 上界扩展实验指标表。
3. 每个仓库的测试集表现。
4. Random Forest 的 Top-20 特征重要性。
5. 特征组消融实验。

推荐消融设置：

| 设置 | 特征组 |
| --- | --- |
| `stats_text` | 代码修改 + 文本 + 仓库 |
| `stats_text_ast` | `stats_text` + AST |
| `stats_text_cfg` | `stats_text` + CFG |
| `main_all` | 代码修改 + 文本 + AST + CFG + 仓库 |
| `upper_bound` | `main_all` + 审查过程特征 |

预期结果解释：

- 如果 `upper_bound` 明显高于 `main_all`，说明审查过程特征确实携带大量后验信息。
- 如果 AST/CFG 提升有限，也可以作为合理结论：patch 片段不完整，传统结构特征表达能力有限。
- 如果 Random Forest 优于 SVM，符合非线性特征和异常值较多的数据特点。

### 6.9 `visualizer.py`

至少生成 10 张图：

## 8. 对后续实验的影响

### 8.1 对实验三的影响

实验三会使用深度学习模型，如 Code2Vec / CodeBERT。实验二必须输出稳定 split，否则实验三结果无法与传统 ML 公平比较。

建议实验三直接复用：

- `human_only_dataset.csv`
- `splits.csv`
- `patch_index.json`
- `features_main.csv` 中的目标列和元信息

### 8.2 对实验四的影响

实验四的大语言模型方法需要上下文构造。实验二的 `patch_index.json` 可以作为 LLM prompt 的代码上下文来源。实验二中对信息泄漏的控制也应延续到实验四。

### 8.3 对实验五的影响

实验五要比较模型在人类代码和 AI 生成代码上的性能。实验二筛掉的 97 条 AI 生成代码 PR 不应删除原始记录，只是不进入人类代码训练集。后续实验五可以直接读取实验一数据集中的 `has_ai_generated_code == True` 样本。

### 8.4 对实验六和实验七的影响

实验六优化 Prompt 和上下文，实验七做 VSCode 插件。实验二的 Random Forest 特征重要性可以告诉后续系统哪些信息最值得展示，例如修改规模、复杂度、测试文件比例、描述完整度等。

## 10. 风险与降级方案

| 风险 | 影响 | 降级方案 |
| --- | --- | --- |
| tree-sitter 安装失败 | 无法生成真实 AST | 使用 lexical AST fallback |
| patch 片段不完整 | AST/CFG 有 ERROR 节点 | 保留错误计数作为特征，报告说明局限 |
| CFG 构建质量不高 | CFG 特征提升有限 | 使用简化 CFG，重点展示覆盖率和消融结果 |
| VSCode 样本减少明显 | 分布偏移 | 按 repo + is_merged 分层，并输出 per-repo 指标 |
| 主实验分数低于上界实验 | 看起来“不够好” | 正面解释为避免后验信息泄漏，主实验更公平 |
| SVM 对异常值敏感 | 指标波动 | 做 99 分位截断和 StandardScaler |
| Random split 学到仓库偏差 | 泛化性不足 | 补充 per-repo 指标；可扩展 leave-one-repo-out |

## 11. 预期结果

不要为了追求极高 Accuracy 使用后验审查字段。合理预期如下：

- Dummy baseline 约等于多数类比例，Accuracy 约 0.64。
- 主实验 SVM / Random Forest 应明显超过 Dummy，目标 Accuracy 约 0.68-0.78，ROC-AUC 约 0.70-0.82。
- Random Forest 通常应优于或接近 SVM，并能给出更清晰的特征重要性。
- 上界扩展实验可能显著高于主实验，因为 review/comment/label 信息和合并结果关系更近。
- 结构特征 AST/CFG 的提升可能有限，但只要覆盖率、样例和消融分析完整，就能满足指导书要求并形成可信结论。

最终报告应强调：

1. 修改规模、代码扰动、PR 描述完整度和结构复杂度会影响合并概率。
2. 传统 ML 的优势是训练成本低、可解释性强。
3. 传统 ML 的局限是依赖人工特征，难以理解深层语义。
4. AST/CFG 特征为后续 Code2Vec、CodeBERT、LLM 实验提供了结构化对照。
